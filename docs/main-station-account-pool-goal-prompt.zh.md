# 主站账号池功能 Codex Goal 提示词

将本文件中“Goal 正文”整体作为 Codex Goal 使用。

## Goal 正文

你正在维护当前工作区中的 UpstreamOps 项目。请完整实现“主站账号池、成本风控与稳定性监控”功能，并把它做成能够实际部署、升级、回滚和持续维护的生产功能。

- 当前工作区：`D:\WorkSpace\upstream-ops`
- 我的仓库：`https://github.com/fausto2022/upstream-ops.git`
- 原始开源项目：`https://github.com/bejix/upstream-ops.git`

先完整阅读需求文档和当前代码，再开始修改。需求文档已经明确的业务关系、状态、保护规则和验收标准不得自行简化。若文档与实际 Sub2API API 或当前项目结构冲突，必须先通过源码、测试或可复现请求确认事实，然后采用兼容设计，并在文档中记录差异和理由。

不要把 `Sub2API Account`、主站用户账号和 UpstreamOps 上游渠道混为一谈。核心业务关系必须保持为：

```text
一个 UpstreamOps 实例只能配置一个 Sub2API 主站。
一个主站账号池可以服务一个或多个主站分组。
一个主站账号池包含多个池成员。
一个池成员 = 上游渠道 + 上游分组 + 一条独立的主站 Account。
一个池成员可以创建托管 Account，也可以绑定已有 Account。
一条主站 Account 只能绑定一个池成员。
实际请求调度由主站 Sub2API 完成，UpstreamOps 负责管理、测活、成本评估和保护。
```

必须遵守以下过程：

1. 检查 `git status`，保留工作区中已有改动，不覆盖、不回退用户修改。
2. 阅读仓库内规范、README、数据库模型、路由、scheduler、syncer、connector、notify、前端路由和现有测试。
3. 为本 Goal 建立阶段计划，并在每个阶段完成后立即更新状态。
4. 先补齐可测试的领域模型和后端能力，再接前端；不要先做只有静态页面的假功能。
5. 每个阶段都必须包含数据库迁移、API、业务逻辑、前端交互、错误处理和测试。
6. 每完成一块就运行对应的最小测试；全部完成后再运行全量验证。
7. 遇到失败要定位根因并修复，不允许为了通过测试删除测试、屏蔽错误或降低关键验收标准。
8. 不执行 `git reset --hard`、强制推送、删除用户数据或其他不可逆操作。
9. 未经用户明确要求，不提交、不 push、不创建 release，不向真实主站写入测试账号。

### 三、强制工程约束

- 保持现有 Go、Gin、GORM、React、TypeScript、Vite、Tailwind 和 Radix UI 技术栈。
- 优先复用现有 repository、service、connector、scheduler、notification、加密和 API 响应模式。
- 不做与需求无关的全量重构、格式化、依赖升级或模块路径改名。
- 不因为仓库属于 `fausto2022` 就贸然修改 `go.mod` module；只有确认所有 import、构建和发布影响后才能处理。
- 不无故新增依赖。确需新增时说明原因、替代方案和包体积/维护影响。
- SQLite 和 MySQL 必须同时兼容；迁移必须幂等，并保留现有数据。
- 金额、倍率、利润率禁止用可能产生业务误判的裸 `float` 比较；采用项目已有精度方案，或实现明确的定点/Decimal 语义和误差边界。
- 所有外部 HTTP 请求必须设置超时、限制响应体大小、校验 URL，并对日志和错误内容脱敏。
- Admin API Key、密码、Cookie、Token 和上游 API Key 必须加密保存或复用现有加密存储，绝不能返回到普通 API、日志、通知和审计中。
- 后端错误必须保留上下文；前端异步错误必须向用户展示，不静默吞掉。
- 自动任务必须支持并发控制、幂等、随机抖动和服务重启后的错峰恢复。
- 真实远端写操作前必须重读绑定、锁和 Account 状态；HTTP 超时后先查询远端最终状态，不盲目重复创建。

### 四、实施阶段

#### 阶段 0：基线调查和技术设计

完成以下调查后再进入业务实现：

- 画出当前 `UpstreamSyncTarget`、`UpstreamSyncGroup`、`UpstreamSyncAccount`、`UpstreamSyncManagedAccount` 的调用和数据关系。
- 找出同步器直接写 `schedulable=true` 的所有路径，以及 Account 创建、更新、测试、删除路径。
- 确认当前数据库迁移入口、SQLite/MySQL 差异、scheduler 注册方式和通知去抖方式。
- 确认前端路由、导航、API client、表单、表格、Dialog、Toast 和 i18n 方式。
- 核对当前官方 Sub2API 源码中的管理员接口、请求字段、分页格式、倍率字段、Account test 行为和 `schedulable` 写法。
- 将关键事实、兼容决策和拟改文件写入简短技术设计文档，避免实现过程中口径漂移。

阶段 0 产出：技术设计、数据迁移方案、API 契约、关键风险清单和细化任务列表。

#### 阶段 1：唯一主站、账号池和显式绑定

实现：

- 唯一 Sub2API 主站配置，复用现有目标站点及 AES-GCM 密钥能力。
- 主站连接测试、分组完整分页同步、Account 完整分页同步和快照更新。
- 主站账号池、池与主站分组关系、池成员、绑定状态和 ownership mode。
- 托管成员：创建或复用源 API Key，在主站创建独立 Account，并保存精确映射。
- 绑定成员：从未占用 Account 中选择，绑定上游渠道和分组；无法验证完整 API Key 时要求人工确认。
- 一条主站 Account 只能被一个成员绑定；数据库约束和 service 校验都要有。
- Account 远端消失后标记 `orphaned`，停止自动写入并通知，保留历史。
- 删除托管成员和绑定成员采用不同语义；默认不删除远端资源。
- 将现有同步分组兼容映射或幂等迁移为账号池，迁移前后数据可核对，不删除旧表。
- 新增“主站”一级导航、主站配置、池列表、池详情、添加成员、绑定已有 Account 等完整页面。

阶段 1 默认只读展示成本，不允许自动利润停用。

阶段 1 验收：需求文档第 26.1、26.2 节全部通过，并有后端测试覆盖分页、唯一绑定、迁移、远端消失和删除语义。

#### 阶段 2：低 Token 测活和稳定性统计

严格按需求文档实现 L0、L1、L2：

- L0 优先使用 billing/models 等零生成接口。
- L1 使用管理员指定的低成本模型，并强制设置协议对应的最小输出上限。
- L2 通过主站管理员接口测试指定 Account，低频运行。
- 不允许把模型列表第一项自动当作稳定测试模型。
- 输出限制不兼容属于测试配置异常，不得直接判为账号故障。
- 记录每次结果、错误分类、HTTP 状态、延迟、Token、估算成本和触发动作。
- 实现连续成功/失败、最近 20 次成功率、1h/24h/7d 成功率、P50/P95 延迟。
- 实现健康状态机、恢复探测、观察模式、预算、并发、最短间隔和随机抖动。
- 401、403、402、429、Timeout、5xx、空响应和模型不兼容必须分类处理。
- 页面支持查看测活记录、趋势、错误摘要、手工 L0/L1/L2 和预算使用量。

测试必须使用 `httptest` 或等价 mock server，验证真实请求体中存在输出上限，同时保证测试不会访问真实上游。

阶段 2 验收：需求文档第 26.3 节全部通过，且 scheduler 重启不会同时补跑所有遗漏任务。

#### 阶段 3：成本核算、利润风控和统一调度锁

实现：

- 按需求文档优先级解析成本来源，保存来源、采集时间和过期时间。
- 同步主站基础倍率、高峰倍率和可读取的用户专属倍率。
- 按“池成员 × 主站分组”生成利润快照。
- 支持成本修正系数、手工覆盖和明确的单位换算。
- 订阅、图片、视频、按次、阶梯计费或数据不完整时返回 `unsupported/unknown`，不得自动停用。
- 实现 `manual`、`margin`、`health`、`sync`、`credential`、`binding` 独立持久化调度锁。
- 每类锁只能管理自身状态，不允许健康恢复清除利润锁，也不允许同步器清除人工锁。
- 实现唯一的调度决策服务，所有 `schedulable` 写入必须经过该服务。
- 清理同步器和其他代码中绕过统一决策直接启用 Account 的路径。
- 实现利润风险、恢复、池容量不足和 critical 通知及去抖。
- 页面展示销售倍率、成本倍率、来源、新鲜度、利润率、风险原因和全部活动锁。

阶段 3 先以观察模式上线。即使计算出风险，也只记录和通知，管理员显式开启自动保护前不得停用 Account。

阶段 3 验收：需求文档第 26.4、26.5 节全部通过，重点增加多锁并发、过期数据、未知计费、远端写失败和同步器误恢复的回归测试。

#### 阶段 4：自动保护、恢复、审计和运维闭环

实现：

- 自动利润停用、自动健康停用和可选自动恢复开关，默认值必须符合需求文档。
- 自动保护启用前必须完成至少一次只读评估，并在 UI 展示预计影响范围。
- 利润风险连续确认、健康失败阈值、恢复阈值和所有锁检查。
- 远端写入前后状态、判断证据、操作结果和失败原因的完整审计。
- 池最小健康成员数、最小有效并发、degraded/critical 计算和通知。
- 手工暂停、解除指定锁、解除全部自动锁、批量检测和批量恢复的保护交互。
- 一键切换回只通知模式；关闭自动保护不能偷偷清除已有锁，必须由管理员确认处理。
- 管理页面补齐 loading、empty、error、partial failure、disabled、orphaned 和权限失败状态。

阶段 4 验收：自动操作可解释、可审计、可回退，且任何单模块恢复都不能错误启用仍被其他原因锁定的 Account。

### 五、后端完成标准

- 数据模型、迁移、repository/service/API 分层符合当前项目风格。
- 所有列表接口支持分页、过滤和稳定排序。
- 所有写接口校验资源归属、绑定冲突和当前状态。
- 主站只能存在一条，数据库和业务层共同保证。
- 调度锁状态变化和远端调度写入具备幂等、串行化和审计证据。
- scheduler 任务支持 context、超时、并发上限和优雅停止。
- 通知不包含秘密，去抖键至少包含事件、池、成员和主站分组。
- API 返回结构、状态码和错误格式与现有接口一致。
- 对关键服务提供单元测试，对 Sub2API 交互提供 mock 集成测试。

### 六、前端完成标准

- 危险操作有影响摘要和二次确认；普通状态图标有 tooltip。
- 长名称、错误内容、倍率来源在桌面和移动端不能溢出或遮挡。

### 八、测试重点

至少覆盖以下高风险场景：

1. 多次执行迁移不重复创建池和成员。
2. 旧同步分组中存在多个目标站点时不擅自选择主站。
3. 超过一页的主站 Group 和 Account 均能完整同步。
4. 两个成员不能绑定同一远端 Account。
5. 绑定成员删除时不删除远端 Account。
6. 远端 Account 消失后变为 `orphaned` 并产生一次通知。
7. L1 请求始终携带受控输出上限。
8. 429 只进入冷却，不被永久判为凭据失效。
9. 成本快照过期、来源不可靠或计费类型不支持时不自动停用。
10. `health` 锁恢复后仍有 `margin` 或 `manual` 锁时，Account 保持不可调度。
11. 同步任务不会重新启用存在任意活动锁的 Account。
12. 远端 schedulable 写入超时后先查询最终状态，不重复执行破坏性操作。
13. 自动保护默认关闭，升级旧数据库后也保持关闭。
14. 所有日志、通知、审计和 API 响应均不泄漏密钥。
15. 容器升级后数据、绑定、锁、策略和审计均保留。

### 九、范围控制

本 Goal 不要求：

- 修改 Sub2API 源码或自行实现请求代理。
- 让一条 Sub2API Account 保存多个上游 API Key。
- 接管 Sub2API 的请求级调度算法。
- 在无法准确计算订阅、图片、视频或复杂阶梯价时强行自动止损。
- 自动 push、强制覆盖远端分支、发布镜像或创建 Release。

不要为了赶进度把“托管 Account”“绑定已有 Account”“统一调度锁”合并成一个模糊状态。这三个概念是本功能避免误操作的关键边界。
